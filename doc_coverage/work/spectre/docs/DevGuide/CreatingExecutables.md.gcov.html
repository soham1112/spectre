       5             : # Creating Executables {#dev_guide_creating_executables}
       6             : 
       7             : There are several different types of executables that can be built:
       8             : - An executable that uses Charm++ to run in parallel,
       9             :   e.g. `Executables/ParallelInfo`
      10             : - An executable that does not use Charm++, does not run in parallel, and
      11             :   supplies its own `main`, e.g. `Executables/Benchmark`
      12             : - An executable that uses Charm++ to run in parallel but supplies its own `main`
      13             : - An executable that uses custom compilation or linking flags,
      14             :   e.g. `DebugPreprocessor`
      15             : - Executables used for evolutions or elliptic solves
      16             : 
      17             : ### Executable Using Charm++ for Parallelization
      18             : 
      19             : A simple example of an executable using Charm++ for parallelization is in
      20             : `src/Executables/Examples/HelloWorld`. To add an executable as a target, use
      21             : the CMake function `add_spectre_parallel_executable` which takes as its four
      22             : arguments
      23             : - the name of the executable
      24             : - the name of the path relative to src that contains the header file (which is
      25             :   the name of the executable with extension `.hpp`) that will define structs and
      26             :   functions to be described below
      27             : - the name of the metavariables struct described below
      28             : - the list of SpECTRE libraries needed for the executable to link
      29             : 
      30             : For example,
      31             : 
      32             : ```
      33             : set(LIBS_TO_LINK
      34             :   Informer
      35             :   Utilities
      36             :   )
      37             : 
      38             : add_spectre_parallel_executable(
      39             :   SingletonHelloWorld
      40             :   Executables/Examples/HelloWorld
      41             :   Metavars
      42             :   "${LIBS_TO_LINK}"
      43             :   )
      44             : ```
      45             : will create the executable `SingletonHelloWorld` from
      46             : SingletonHelloWorld.hpp which is found in the directory
      47             : `src/Executables/Examples/HelloWorld`.  `Metavars` will be used as the
      48             : metavariables struct, and `SingletonHelloWorld` will link against the
      49             : `%Informer` and `Utilities` SpECTRE libraries.
      50             : 
      51             : The header file from which an executable is generated must define a
      52             : metavariables struct that can be thought of as a compile-time input file that
      53             : defines what the executable will do.  The `Metavars` struct for
      54             : `SingletonHelloWorld` is a minimal example of a metavariables struct.
      55             : 
      56             : \snippet SingletonHelloWorld.hpp executable_example_metavariables
      57             : 
      58             : Each metavariables must define an enum class `Phase` with the phases of the
      59             : executable (which must include `Initialization` and `Exit`), and a static
      60             : function `determine_next_phase`.  In the example, the only additional phase is
      61             : `Execute`, and the phases are executed in order.  Charm++ will execute each
      62             : phase until it detects that nothing is happening (quiescence detection).  As
      63             : this represents a global synchronization point, the number of phases should be
      64             : minimized in order to exploit the power of SpECTRE.
      65             : 
      66             : Each metavariables must define a type alias `component_list` that is a
      67             : `tmpl::list` of the parallel components used by the executable.
      68             : `SingletonHelloWorld` defines a single component `HelloWorld`
      69             : 
      70             : \snippet SingletonHelloWorld.hpp executable_example_singleton
      71             : 
      72             : which specifies via the `chare_type` type alias that it is a singleton parallel
      73             : component which means that only one such object will exist across all processors
      74             : used by the executable.  Each component must define the static functions
      75             : `initialize`, which is executed during the `Initialization` phase, and
      76             : `execute_next_phase` which is executed during the phases (other than
      77             : `Initialization` and `Exit`) defined in the metavariables struct.  In
      78             : `SingletonHelloWorld`, nothing is done during the initialization phase, while
      79             : the `PrintMessage` action is called during the `Execute` phase.
      80             : 
      81             : \snippet  SingletonHelloWorld.hpp executable_example_action
      82             : 
      83             : The `PrintMessage` action is executed on whatever process the singleton
      84             : component is created upon, and prints a message.
      85             : 
      86             : Executables can read in an input file (specified by the `--input-file` argument)
      87             : that will be parsed when the executable begins.  %Options specified in the input
      88             : file can be used to either place items in a constant global cache (by specifying
      89             : options in the `const_global_cache_tag_list` of the metavariables or component
      90             : structs) or be passed to the `initialize` function of a component (by specifying
      91             : options in the `options` type alias of the component).  `SingletonHelloWorld`
      92             : specifies a single option
      93             : 
      94             : \snippet SingletonHelloWorld.hpp executable_example_options
      95             : 
      96             : which a string specifying a name that will be placed into the constant global
      97             : cache.  The string is fetched when performing the `PrintMessage` action. Items
      98             : in the constant global cache are stored once per node that the executable runs
      99             : on. An example input file for `SingletonHelloWorld` can be found in
     100             : `tests/InputFiles/ExampleExecutables/SingletonHelloWorld.yaml` and shows how to
     101             : specify the options (lines beginning with a `#` are comments and can be
     102             : ignored).
     103             : 
     104             : In addition to defining the metavaribles and component structs, the header for
     105             : an executable must define two lists of functions
     106             : 
     107             : \snippet  SingletonHelloWorld.hpp executable_example_charm_init
     108             : 
     109             : that are executed at startup by Charm++ on each node and processor the
     110             : executable runs on.
     111             : 
     112             : Furthermore among the included header files
     113             : 
     114             : \snippet  SingletonHelloWorld.hpp executable_example_includes
     115             : 
     116             : must be the appropriate header for each parallel component type, which in the
     117             : `SingletonHelloWorld` example is `AlgorithmSingleton.hpp`.  Note that
     118             : these headers are not in the source tree, but are generated automatically when
     119             : the code is compiled.
     120             : 
     121             : See [the Parallelization documentation](group__ParallelGroup.html#details)
     122             : for more details.
     123             : 
     124             : ### Executable Using Charm++ with Custom main()
     125             : 
     126             : While this is technically possible, it has not been tested. We recommend using
     127             : the Charm++ supplied main chare mechanism for the time being.
     128             : 
     129             : ### Executable Not Using Charm++
     130             : 
     131             : An example of an executable that does not use Charm++ for parallelization but
     132             : still can use all other infrastructure in SpECTRE is in
     133             : `src/Executables/Examples/HelloWorldNoCharm`. Adding a non-Charm++ executable to
     134             : SpECTRE mostly follows the standard way of adding an executable using CMake. The
     135             : only deviation is that the `CMakeLists.txt` file must tell Charm++ not to add a
     136             : `main()` by passing the link flags `-nomain-module -nomain`. This is done using
     137             : CMake's `set_target_properties`:
     138             : 
     139             : ```
     140             : set_target_properties(
     141             :   ${EXECUTABLE}
     142             :   PROPERTIES LINK_FLAGS "-nomain-module -nomain"
     143             :   )
     144             : ```
     145             : 
     146             : To add the executable as a target you must use the `add_spectre_executable`
     147             : function, which is a light weight wrapper around CMake's `add_executable`.
     148             : For example,
     149             : 
     150             : ```
     151             : add_spectre_executable(
     152             :   ${EXECUTABLE}
     153             :   EXCLUDE_FROM_ALL # Exclude from calls to `make` without a specified target
     154             :   HelloWorld.cpp
     155             :   )
     156             : ```
     157             : 
     158             : You can link in any of the SpECTRE libraries by adding them to the
     159             : `target_link_libraries`, for example:
     160             : 
     161             : ```
     162             : target_link_libraries(
     163             :   ${EXECUTABLE}
     164             :   DataStructures
     165             :   )
     166             : ```
     167             : 
     168             : We recommend that you add a test that the executable properly runs by adding an
     169             : input file to `tests/InputFiles` in an appropriate subdirectory. See
     170             : [`tests/InputFiles/ExampleExecutables/HelloWorldNoCharm.yaml`]
     171             : (https://github.com/sxs-collaboration/spectre/tree/develop/tests/InputFiles/
     172             : ExampleExecutables/HelloWorldNoCharm.yaml)
     173             : for an example.
     174             : The input file is passed to the executable using `--input-file
     175             : path/to/Input.yaml`. In the case of the executable not taking any input file
     176             : this is just used to generate a test that runs the executable.
     177             : 
     178             : For these types of executables `main` can take the usual `(int argc, char
     179             : *argv[])` and parse command line options. Executables not using Charm++ are just
     180             : standard executables that can link in any of the libraries in SpECTRE.
     181             : 
     182             : \warning
     183             : Currently calling `Parallel::abort` results in a segfault deep inside Charm++
     184             : code. However, the error messages from `ASSERT` and `ERROR` are still printed.
     185             : 
     186             : ### Executable With Custom Compilation or Linking Flags
     187             : 
     188             : Use the CMake function `set_target_properties` to add flags to an executable. To
     189             : call a completely custom compiler invocation you should use the
     190             : `add_custom_target` CMake function. The need for the `custom_target` level of
     191             : control is rare and should generally be avoided since it adds quite a bit of
     192             : technical debt to the code base. Thus, it is not explained here. If you are
     193             : certain you need it you can see the `DebugPreprocessor` executable's
     194             : `CMakeLists.txt` file for an example.
     195             : 
     196             : ### Executable Used for Evolution or Elliptic Solve
     197             : 
     198             : Once they are written, see the tutorials specific to evolution and elliptic
     199             : solves.
